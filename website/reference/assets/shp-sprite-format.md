# SHP shape (sprite) format

*Last verified: 2026-07-21. Version coverage: **Command & Conquer: Yuri's Revenge** only — header, frame directory, both frame decoders, and the draw path are reversed and oracle-tested there. This is the shared TS-era SHP variant used across Tiberian Sun, Red Alert 2, and Yuri's Revenge; **Red Alert 2 is expected to be byte-for-byte identical** (it is generated by the same accessor code) but has not been separately confirmed, and the **Tiberian Sun** comparison is pending. Both are stated as expected, not verified.*

SHP is the engine's paletted-sprite container. Infantry, projectiles, UI pips, cameos, and many animations are stored as SHP files: a single canvas size, a directory of frames, and per-frame pixel data that is either raw palette indices or a run-length-encoded scanline stream. Every pixel is one 8-bit index into the active palette; there is no color data in the file itself beyond a per-frame average color that the engine does not use for drawing.

This entry describes the **TS-era ("TS SHP") variant** — the format shipped by Tiberian Sun and inherited unchanged by Red Alert 2 and Yuri's Revenge. It is a different layout from the older Tiberian Dawn / Red Alert 1 shape format.

## On-disk layout

A SHP file is a short header, a frame directory, then pixel data:

```
byte 0   u16  zero        always 0 in a file
byte 2   i16  Width       full-frame canvas width
byte 4   i16  Height      full-frame canvas height
byte 6   i16  FrameCount
byte 8   FrameCount x 24-byte frame entries
...      pixel data, located by each entry's data offset
```

The leading `u16` is `0` in every on-disk file. (A value with all bits set, 65535, only ever appears in memory, marking a lazy-load wrapper the engine builds at runtime; it is never written to a file.)

There is **no magic number, signature, or size field** anywhere in the format, and the engine performs no such validation when loading — it reads the header, trusts the frame count, and indexes the directory directly. Files are self-describing only through `Width`, `Height`, and `FrameCount`.

### Frame directory entry (24 bytes)

Each frame is described by a fixed 24-byte record. Pixel data begins immediately after the directory (at byte `8 + FrameCount x 24`) and each entry points into it:

```
byte 0   i16  X            frame's left edge within the canvas
byte 2   i16  Y            frame's top edge within the canvas
byte 4   i16  Width        this frame's sub-rectangle width
byte 6   i16  Height       this frame's sub-rectangle height
byte 8   u32  Flags        only the compressed bit (mask value 2) is read
byte 12  u8   R            average color, red
byte 13  u8   G            average color, green
byte 14  u8   B            average color, blue
byte 15  u8   (padding, never read)
byte 16  u32  (never read by any drawing consumer)
byte 20  i32  DataOffset   byte offset from the START of the file; 0 = empty frame
```

A frame carries its **own** `Width`/`Height` and an `X`/`Y` origin, distinct from the file-level canvas. This lets a frame store only its non-empty sub-rectangle and be composited at the right place on the larger canvas — the common case for animation frames that only differ in a small region.

The average-color triplet is retrievable but is **not** consulted by the normal blit path; it is metadata a tool can read, not something the sprite draw uses.

### Flags

`Flags` is a 32-bit field, but the drawing engine reads exactly **one bit**: the compressed bit (mask value **2**). When set, the frame's pixels are RLE-zero scanlines; when clear, they are raw palette indices.

Stock files are observed carrying `Flags` values of 0, 1, 3, and 4. Of the other bits:

- **Bit value 1** ("has transparency", by community convention) has **no consumer** in the draw path. Transparency does not depend on it — the engine treats palette index 0 as transparent unconditionally (see below). It is encoder-informational only.
- **Bit value 4** was observed only on an empty frame. It is likewise not read by the draw path.

In other words: the only flag that changes how a frame is decoded is the compressed bit. The rest are inert as far as rendering is concerned.

## Transparency

**Palette index 0 is always transparent**, in both frame encodings, regardless of any flag. Uncompressed frames skip index-0 pixels when blitting; compressed frames encode transparency as explicit zero-runs (below) and never write those pixels. There is no separate "transparent color" field and no flag that toggles this behavior.

## Compressed frames: RLE-zero scanlines

When the compressed bit is set, the frame's data is a sequence of **`Height` scanline records**, one per row:

```
[u16 recordLength (little-endian, INCLUDES the 2 length bytes)] [recordLength - 2 payload bytes]
```

The record length exists **only to hop from one row to the next** — the engine advances to the next scanline by adding the current record's length to the current pointer. This is also how it skips vertically clipped rows without decoding them.

The payload itself is decoded by a **width-bounded** state machine that never looks at the record length. For a row of the frame's `Width`:

```text
remaining = frameWidth
while remaining > 0:
    v = next payload byte
    if v == 0:
        count = next payload byte
        skip 'count' transparent pixels   (nothing is written)
        remaining -= count
    else:
        write remapped palette index v     (one pixel)
        remaining -= 1
```

Two consequences of this being width-bounded rather than record-bounded are load-bearing and are reproduced faithfully:

- **The decoder stops at `Width` pixels, not at the end of the record.** It reads exactly as many payload bytes as it takes to produce `Width` output pixels. A row whose payload under-fills the width would read into the following record; this is not observed in stock data, but the behavior is defined by the width bound.
- **Zero-run overrun is tolerated.** A zero-run whose count pushes `remaining` below zero simply ends the row — the surplus is discarded rather than causing an error. This happens in real stock data: in the standard `pips.shp`, 10 of the 15 rows of the multiplayer-pip frame encode 16 transparent-terminated pixels on a 15-pixel-wide frame, overrunning the width by one each. The engine absorbs it silently.

Horizontal clipping runs through the **same** state machine: to clip the left edge, the decoder advances the identical byte-consuming logic for the clipped pixel count first (correctly splitting a zero-run that straddles the clip boundary) before it begins writing visible pixels.

## Uncompressed (plain) frames

When the compressed bit is clear, the frame's data is exactly **`Width x Height` raw palette indices**, stored row-major (each row is `Width` bytes, rows stacked top to bottom). Drawing walks the rows directly, and — as with compressed frames — **index 0 is skipped as transparent** while every other index is remapped and written. Transparency here does not depend on any flag bit.

## Empty frames

A frame whose `DataOffset` is **0** is empty — the offset is a null sentinel, not a pointer to a zero-length region. In stock files an empty frame also carries zero geometry (a zero sub-rectangle). The normal draw path never reaches a pixel fetch for such a frame because it bails earlier on the empty clip rectangle. Empty frames are common as placeholders inside otherwise-populated animation sequences.

## What this entry does not claim

- That **Red Alert 2** or **Tiberian Sun** files decode identically at the byte level. RA2 is expected to be identical (same generation path) but is not separately confirmed here; the TS comparison is pending. Only Yuri's Revenge is verified.
- Anything about **palette selection, remap tables, unit-color remapping, lighting, or blitter/blend styles**. Those are presentation concerns layered on top of the decoded index buffer, not part of the SHP container, and are out of scope for this entry.
- The exact meaning of the unread 32-bit field in each directory entry, or of flag bits other than the compressed bit. They are not consumed by the drawing engine; this entry only states that fact, not what an encoder may have intended by them.
- The behavior of a malformed frame that sets `DataOffset` to 0 while also declaring a non-empty rectangle. Such a combination does not occur in stock data, and its outcome in the retail engine is a null dereference rather than a defined result.
